# 화면설계서 작성 프롬프트

## 프롬프트 사용 방법

아래 프롬프트를 AI에게 제공하고, [대괄호] 안의 내용을 실제 프로젝트 정보로 교체하여 사용하세요.

---

## 📋 화면설계서 작성 프롬프트

```
[프로젝트명]의 [기능명] 화면에 대한 상세 설계서를 작성해주세요.

### 📌 기본 정보
- **기능명**: [예: Settings 팝업, 사용자 관리 화면, 데이터 분석 대시보드]
- **대상 사용자**: [예: User/Admin/Master, 일반 사용자/관리자]
- **주요 목적**: [예: 시스템 설정 관리, 사용자 데이터 조회 및 편집]

### 📐 설계서 구조

다음 섹션들을 포함하여 작성해주세요:

#### 1. 문서 정보
- 문서 제목, 버전, 작성일, 작성자
- 문서 목적 및 범위

#### 2. 개요
- 기능 개요 및 목적
- 주요 기능 목록
- 사용자 권한별 기능 차이

#### 3. 화면 구조
- 전체 화면 레이아웃 (팝업/페이지)
- 탭 구조 (있는 경우)
- 서브탭/섹션 구조
- 권한별 화면 구성 차이

#### 4. UI 구성요소 상세
각 UI 요소에 대해 다음 정보 포함:
- **요소명**: [예: Language 설정, 데이터 불러오기 방식]
- **타입**: [드롭다운, 체크박스, 토글, 텍스트 입력, 버튼 등]
- **기본값**: [초기 표시 값]
- **옵션 목록**: (드롭다운/라디오 등의 경우)
- **필수 여부**: [필수/선택/조건부 필수]
- **권한**: [User/Admin/Master 또는 접근 가능 역할]
- **위치**: [화면에서의 배치 위치]

#### 5. 조건부 표시 규칙
- 특정 조건에서만 표시되는 요소
- 활성화/비활성화 조건
- 상호 배타적 옵션 관계
- 연쇄적 표시/숨김 규칙

**표 형식 예시:**
| 요소명 | 표시 조건 | 비고 |
|--------|----------|------|
| [요소 A] | [조건 1]일 때 표시 | [추가 설명] |

#### 6. 검증 규칙 및 경고
- 필수 항목 검증
- 조건부 필수 항목 검증
- 입력값 형식 검증
- 경고 메시지 발생 조건 (구체적인 메시지 텍스트는 제외)
- 검증 실패 시 동작

**경고 메시지 작성 원칙:**
- 구체적인 메시지 텍스트는 기재하지 않음
- "경고 메시지 표시" 또는 "[항목명] 검증 실패 시 경고" 형태로 작성
- 발생 조건과 동작만 명시

#### 7. 인터랙션 및 동작
- 버튼 클릭 시 동작
- 설정 변경 시 동작
- 저장/취소 로직
- 팝업/모달 표시 조건
- 탭 전환 동작

#### 8. 상태 관리
- 변경사항 추적 방법
- "저장되지 않은 변경사항" 경고 표시 조건
- Critical Settings (중요 설정 변경 시 특별 처리)

#### 9. 저장 프로세스
- 저장 버튼 활성화 조건
- 저장 전 검증 단계
- 저장 성공/실패 피드백
- 저장 후 후속 동작 (예: 화면 새로고침, 다른 탭 닫기)

#### 10. 특수 기능 (해당하는 경우)
- Backup/Restore 메커니즘
- Import/Export 기능
- Diff 비교 기능
- 자동 백업 규칙

#### 11. 플로우차트
주요 사용자 시나리오에 대한 플로우차트:
- 화면 열기 플로우
- 설정 변경 및 저장 플로우
- 팝업 닫기 플로우
- 복원/가져오기 플로우

```
[사용자 액션]
    ↓
[시스템 체크/처리]
    ↓
    ├→ [조건 A] → [결과 A]
    └→ [조건 B] → [결과 B]
```

#### 12. 테스트 항목
화면 순서대로 상세 테스트 케이스 작성:

**각 테스트 케이스 포함 사항:**
- TC-[카테고리]-[번호]: 테스트 케이스명
- **권한**: 테스트 대상 권한
- **전제조건**: (필요한 경우)
- **테스트 단계**: 1, 2, 3... 순서대로
- **예상 결과**: 각 단계별 또는 최종 예상 결과

**테스트 카테고리 예시:**
- 기본 기능 테스트 (각 탭/섹션별)
- 권한별 접근 테스트
- 조건부 표시/숨김 테스트
- 검증 규칙 테스트
- 저장/복원 테스트
- 통합 시나리오 테스트

#### 13. 변경 이력
| 버전 | 날짜 | 작성자 | 변경 내용 |
|------|------|--------|----------|
| 1.0 | YYYY-MM-DD | [이름] | 초안 작성 |

### 🎨 스타일 가이드

1. **명확성**: 모호한 표현 피하기, 구체적인 값과 조건 명시
2. **완전성**: 모든 예외 케이스와 엣지 케이스 포함
3. **일관성**: 용어, 형식, 구조의 일관성 유지
4. **시각화**: 복잡한 로직은 플로우차트나 표로 표현
5. **검증 가능성**: 개발자와 QA가 명확히 구현/테스트할 수 있도록 작성

### 📝 작성 시 주의사항

1. **권한 관리**: 각 기능에 대한 권한 정보를 명확히 명시
2. **조건부 로직**: "만약 A이면 B를 표시" 형태로 명확하게 작성
3. **경고 메시지**: 구체적인 메시지 텍스트는 제외하고, 발생 조건과 동작만 명시
4. **상호작용**: 한 요소 변경이 다른 요소에 미치는 영향 명시
5. **테스트 커버리지**: 모든 주요 기능과 엣지 케이스를 테스트 항목에 포함

### 🔍 추가 정보

#### 현재 프로젝트의 특수 요구사항
[여기에 프로젝트별 특수 요구사항 추가]
- 예: GDPR 규정 준수 필요
- 예: 분석 탭 자동 닫기 동작 필요
- 예: 실시간 동기화 지원

#### 기술 스택 및 제약사항
- Frontend: [예: React, Vue, Angular]
- 상태 관리: [예: useState, Redux, Vuex]
- UI 라이브러리: [예: Material-UI, Ant Design]
- 제약사항: [예: 브라우저 호환성, 성능 요구사항]

#### 참고 자료
- 기존 화면 스크린샷: [링크 또는 파일 경로]
- 기획서: [링크]
- API 문서: [링크]
```

## 💡 실제 사용 예시

### 예시 1: Settings 팝업 화면설계서

```
SV Pro의 Settings 팝업 화면에 대한 상세 설계서를 작성해주세요.

### 📌 기본 정보
- **기능명**: Settings 팝업 (시스템 설정 관리)
- **대상 사용자**: User/Admin/Master (3단계 권한)
- **주요 목적**: PCR 분석 시스템의 전반적인 설정 관리, 백업/복원 기능 제공

### 📐 주요 특징
- 6개 탭 구조: General, Display Setting, Export, Print, Assay Pack, Backup/Restore
- 권한별 차등 접근: User는 일부 탭만, Admin은 대부분, Master는 전체 접근
- GDPR 규정 준수를 위한 개인정보 보호 모드
- 실시간 변경사항 추적 및 경고
- Critical Settings 변경 시 분석 탭 자동 닫기
- 자동/수동 백업 및 Diff 비교 복원 기능

### 📝 세부 요구사항
- 조건부 필수 항목 다수 (CSV Header Settings, Logo Location 등)
- 상호 배타적 옵션 (LIMS ↔ CSV, Barcode + Name 중복 불가)
- GDPR 모드에 따른 동적 옵션 표시/숨김
- Backup File List에서 직접 복원 가능
- Import 시 Diff 비교 모달로 변경사항 확인

[위의 전체 구조에 따라 상세 설계서를 작성해주세요]
```

## 🎯 프롬프트 최적화 팁

### 1. 단계적 작성 요청
복잡한 화면의 경우 섹션별로 나누어 작성 요청:

```
먼저 1~3번 섹션(문서 정보, 개요, 화면 구조)만 작성해주세요.
→ 확인 후
다음으로 4~6번 섹션(UI 구성요소, 조건부 규칙, 검증)을 작성해주세요.
→ 확인 후
...
```

### 2. 기존 자료 제공
```
첨부된 스크린샷을 참고하여 화면설계서를 작성해주세요.
또는
기존 기획서의 내용을 바탕으로 더 상세한 화면설계서를 작성해주세요.
```

### 3. 반복 개선
```
작성된 화면설계서에 다음 내용을 추가해주세요:
- [추가할 기능 또는 섹션]

또는

테스트 항목을 더 상세하게 작성해주세요. 각 테스트 케이스에 전제조건과 예상 결과를 명확히 추가해주세요.
```

### 4. 특정 형식 요청
```
조건부 표시 규칙을 표 형식으로 정리해주세요.
또는
저장 프로세스를 플로우차트 형식으로 작성해주세요.
```

## 📚 참고: 좋은 화면설계서의 특징

### ✅ DO
- ✓ 구체적인 값과 조건 명시 ("3초 후", "5개 이상")
- ✓ 모든 예외 케이스 문서화
- ✓ 경고/오류 발생 조건을 명확히 기재 (메시지 텍스트는 제외)
- ✓ 권한별 차이를 명확히 구분
- ✓ 테스트 가능한 수준으로 상세하게 작성
- ✓ 플로우차트와 표를 적극 활용
- ✓ 변경 이력을 꼼꼼히 기록

### ❌ DON'T
- ✗ 모호한 표현 ("적절히", "필요시", "잘")
- ✗ 암묵적 가정 ("당연히 ~할 것이다")
- ✗ 예외 케이스 누락
- ✗ 권한 정보 누락
- ✗ 테스트하기 어려운 추상적 설명
- ✗ 일관성 없는 용어 사용
- ✗ 변경사항 추적 부재
- ✗ 구체적인 경고 메시지 텍스트 포함
